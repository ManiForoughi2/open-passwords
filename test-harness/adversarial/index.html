Adversarial test pages — NO autofill expected

Real-world NON-login text inputs. These verify the password-manager extension does NOT show its autofill dropdown where it shouldn't. They reproduce the reported bug where the extension wrongly appeared on an Instagram "Tag: [Search]" box.

On every page below (except where noted), the correct behavior is: NO autofill dropdown should appear on the input. The one exception is mixed-page.html, which deliberately also contains a REAL login form where the dropdown SHOULD appear.
Site search bar NO DROPDOWN
search-bar.html

A formless type="search" plus one inside <form role="search">. The commonest non-login field.
Social "Tag people" picker NO DROPDOWN
social-tag.html

The literal repro: formless placeholder="Search" / aria-label="Tag people", plus an ARIA combobox variant.
Compose / DM box NO DROPDOWN
compose-message.html

Chat send input, a <textarea>, and a contenteditable composer.
Comment / reply field NO DROPDOWN
comment-box.html

An name="comment" add-a-comment input and a name="reply" reply field.
Checkout / shipping address NO DROPDOWN
checkout-address.html

Name/address/city/postal/email with autocomplete address tokens but NO password — address autofill, not login.
Newsletter signup NO DROPDOWN
newsletter.html

A lone email field in a <form autocomplete="off"> — subscription, not login.
Edit profile settings NO DROPDOWN
profile-edit.html

Name/email/phone settings form with NO password — editing profile, not logging in.
Mixed page: search + real login SPLIT BEHAVIOR
mixed-page.html

A header search box (NO dropdown) AND a real username+password login form lower down (dropdown SHOULD appear). The discrimination test.

See NOTES.md in this folder for the per-page list of exact selectors that must show NO dropdown (and, for the mixed page, which ones SHOULD).
