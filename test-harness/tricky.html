Scenario: Edge cases on one page. Each card below is a known hard case for autofill detection. Watch which fields the extension decorates / offers to fill.

1. Login inside an <iframe> iframe

A real login form lives in a separate document loaded via iframe (iframe-pages/iframe-login.html). NOTE: this is a same-origin iframe because we can't host a real cross-origin frame locally. Real cross-origin iframes (e.g. embedded SSO/checkout) are stricter and some managers refuse to fill them. Correct behavior: the extension should detect the username/password inside the frame and still offer to fill it.

2. Hidden password field hidden

There is a real type="password" field on this card, hidden with display:none. Correct behavior: a display:none field is a legitimate hidden-then-revealed pattern, so the extension SHOULD fill it (Apple and Chrome do). It still must NOT fill the clickjacking pattern (offscreen / 1px / opacity:0) — see the dedicated Clickjacking test for that.

3. Password field with no name and no id no-name

A password input with no name, no id, and no autocomplete attribute - the manager must rely on type/context heuristics alone. Correct behavior: ideally still recognized as a password field and offered a fill, paired with the username above it.

4. Dynamically-injected login form injecting in 1.5s...

The login form below is NOT in the initial HTML. It is created by JavaScript and inserted into the DOM after a 1.5s delay (a common SPA / lazy-render pattern). Correct behavior: the extension must use a MutationObserver (or re-scan) and detect/decorate this form once it appears - not only fields present at page load.
